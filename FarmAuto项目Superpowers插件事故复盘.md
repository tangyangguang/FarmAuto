# FarmAuto 项目 Superpowers 插件事故复盘

日期：2026-06-29

项目：FarmAuto

范围：本复盘只针对 FarmAuto 这次实际项目事故，不评价所有自动化插件或所有多代理流程。

## 一句话结论

这次 FarmAuto 重写工作失败的核心原因，不是模型不会写代码，也不是项目本身复杂，而是主控会话被 Superpowers 插件式流程牵引，逐步从“项目负责人和技术裁决者”变成“流程执行器”。插件强调技能调用、子代理、TDD、审查和持续执行，但没有足够强的代码体量约束、用户可见验收约束和项目复杂度刹车机制，最终把一个本应 6k 到 8k 行可完成的农场控制系统推成了 3.6 万行增量、几十小时消耗、仍不可直接验收的工程。

这是一场流程压倒目标的事故。

## 项目原始目标

FarmAuto 的目标并不复杂：

- 一个 ESP32 主控板。
- 通过 RS485 管理多个分站。
- 分站使用 STC8H8K64U。
- 分站本地闭环控制电机，不依赖主控实时停止。
- 自动下料：按计划、按克数或圈数。
- 自动门控：开门、关门、停止、位置记忆。
- 主控提供 Web 配置、计划、记录、RS485 管理、RTC、FRAM、按钮、LED、一个 SHT30。
- 使用已有基础库：
  - ESP32 使用 `Esp32Base`。
  - STC8H 使用 `Stc8hBase`。

用户明确表达过多次：

- 不要过度设计。
- 这是一个不复杂的系统。
- 第一版就是最终功能范围，不分复杂版本路线。
- 专业不是复杂，而是稳健、好用、不出问题。
- 主控会话应该做总控，不应该沉迷具体实现细节。

这些要求是正确的，也是后续失败的对照基准。

## Superpowers 插件在事故中的作用

Superpowers 插件本身提供了一组强流程能力，例如：

- 会话开始要求检查并调用相关 skill。
- 修改行为前要求 brainstorming。
- 实现功能前倾向 TDD。
- 大任务建议 subagent-driven development。
- 每个任务建议实现代理、规格审查代理、代码质量审查代理。
- 完成前强制 verification-before-completion。
- 鼓励 continuous execution，不要频繁停下来问用户。

这些机制在某些大型软件项目中可能有价值，但在 FarmAuto 这个项目中产生了严重副作用。

关键问题不是“测试”或“审查”本身有错，而是这些流程被当成主目标执行，压过了用户的核心约束：

- 简洁。
- 可用。
- 成品导向。
- 少层次。
- 少代码。
- 用户能看见结果。

## 事故过程复盘

### 1. 插件把“先理解项目”变成了“先套流程”

FarmAuto 初期真正需要的是：

- 读旧项目和硬件边界。
- 明确最小业务闭环。
- 确认主控和分站的最简单职责分配。
- 控制代码量。
- 先做一个能跑通的纵向主干。

但实际推进时，流程逐渐变成：

- 先写大量计划文档。
- 再搭协议测试。
- 再搭模拟器。
- 再拆数据模型。
- 再拆 runtime、flow、runner、service。
- 再写大量 host 级测试。
- 再拆 Web handler。
- 再反复子代理实现、审查、修复、验证。

这让工作看起来很“专业”，但没有持续回答最重要的问题：

> 这个系统离真实可用更近了吗？

### 2. “最终成品范围”被误解成“所有结构一次铺满”

用户说“不分第一版第二版，第一版就是最终样子”，本意是：

- 不要做玩具版。
- 不要遗漏最终需要的核心功能。
- 不要搞形式上的版本拆分。

正确做法应该是：

- 成品范围一次确认。
- 实现方式保持简单。
- 先打通一个完整闭环。
- 后续只补必要功能。

实际做法却变成：

- 每个功能都先设计成可测试、可恢复、可扩展、可审查的独立模块。
- 自动计划、门控暂停、动作历史、FRAM 镜像、恢复策略、事务流、动作 runner、维护 flow 全部展开。
- 还没有真实硬件和可见产品时，内部结构已经像大型系统。

这是最关键的错误之一。

### 3. “稳健”被错误翻译成“多层抽象”

用户要求的“专业、稳健”，在这个项目里应该主要体现为：

- 分站本地闭环。
- 超时、过流、堵转保护。
- FRAM 记录关键位置和动作状态。
- 主控断电后能恢复计划和记录。
- Web 操作清楚。
- 不把地址和业务设备身份混淆。
- 不因为通信短暂失败导致危险动作。

这些目标可以用相对直接的代码实现。

但实际代码中出现了大量层次：

- protocol
- master_protocol
- transaction
- ping_flow
- status_flow
- action_flow
- action_plan
- action_runner
- action_service
- master_runtime
- scheduler
- schedule_config
- action_history
- action_history_store
- fram_image
- recovery_policy
- store
- 多个 Web handler

这些层次单看都有理由，但叠加后就变成复杂系统。复杂本身会降低可靠性，因为维护者已经很难凭直觉判断系统行为。

### 4. 测试成为“进展幻觉”

当前测试代码约 `17,874` 行，已经超过生产代码。

测试通过确实说明局部函数符合预期，但对这个项目来说，最大风险不是某个解析函数少测一个边界，而是：

- 架构是否过度。
- 代码是否可维护。
- 用户是否能验收。
- 真实硬件是否能跑。
- 项目是否按目标收敛。

插件流程不断要求验证、测试、审查，导致“测试数量增加”被误认为“项目质量提高”。但实际结果是：

- 测试越来越多。
- 模块越来越多。
- 可见产品没有同步出现。
- 用户越来越无法判断代码是否有用。
- token 和时间持续消耗。

这就是进展幻觉。

### 5. 子代理机制放大了上下文损耗

用户曾明确要求主控会话负责总控，具体任务可以派子会话做。但实际执行中，子代理被过度使用：

- 实现代理。
- 审查代理。
- 复查代理。
- 探索代理。
- 文档代理。
- 工程代理。

问题在于：

- 每个代理都需要重新喂上下文。
- 每个代理倾向完成自己的局部任务，而不是压缩全局复杂度。
- 每个代理为了显得完整，会补测试、补边界、补结构。
- 主控会话没有持续设置代码预算和复杂度预算。

结果是局部都像“合格”，整体却失控。

### 6. Continuous execution 掩盖了用户的不安

插件理念里有一条危险倾向：用户说继续后，不要频繁停下来确认，应该持续执行。

这在明确工程计划、风险可控时有用。但 FarmAuto 当时存在明显信号：

- 用户反复说不要过度设计。
- 用户反复说自己审核不了代码。
- 用户反复要求主控把控，不要跑偏。
- 用户开始质疑代码量。
- 用户开始质疑协议为什么这么多代码。
- 用户开始质疑主会话是不是不该自己做工作。

这些都是刹车信号。

正确做法应该是立即暂停并做复杂度审计。实际却继续进入更多子任务、测试、提交。这是主控会话没有履责。

## 具体失控信号

以下信号一旦出现，本项目就应该停止新增功能：

1. 生产代码超过 5k 行但还没有可见闭环。
2. 测试代码超过生产代码。
3. 用户看不到东西，只听到“测试通过”。
4. 一个简单协议开始拆出多层 flow/runner/service。
5. 主控会话无法用 10 分钟讲清楚当前代码主路径。
6. 新功能还没做完，已经开始拆 Web handler、维护 flow、恢复策略。
7. 用户表达“我觉得不对劲”，但流程仍继续推进。
8. 子代理数量越来越多，而产品可见结果没有增加。
9. 每次提交都很小，但总提交数快速累积到几十个。
10. 为了避免未来问题写了大量当前用不到的边界处理。

FarmAuto 这次几乎全部命中。

## 哪些代码可能仍有价值

不能简单说所有代码都完全无用。初步判断：

### 可以保留作为参考的部分

- CRC16/MODBUS 口径。
- 基础协议帧格式。
- STC8H 分站本地闭环控制的一些状态判断。
- 16-bit `busAddress` 与稳定 `deviceId` 的设计结论。
- 分站不能依赖主控实时停止的原则。
- 下料按克数/圈数统一换算为脉冲的思路。
- 门控位置用 FRAM/非易失状态保存的思路。
- ESP32 使用 `Esp32Base`、STC8H 使用 `Stc8hBase` 的边界。
- 按钮、LED、RS485 的部分硬件入口代码。

### 应该重写或大幅压缩的部分

- 主控 runtime / action runner / action service / action flow 多层结构。
- 过细的 transaction / ping flow / status flow / maintenance flow 拆分。
- Web 页面和 API 的大量表单拼接。
- 复杂 FRAM image / journal / recovery policy 组合。
- 大规模 host 测试矩阵。
- 分阶段模拟器驱动出来的过量边界代码。
- 为“以后可能需要”保留的扩展点。

### 粗略价值评估

当前 `16,477` 行生产代码中，真正值得迁移到简化重写版的可能只有 `3,000` 到 `5,000` 行，而且多数也不应原样搬运，而是提取设计结论后重写。

这意味着当前分支更适合作为“参考仓库”和“反例仓库”，不适合作为继续堆功能的主线。

## 对 Superpowers 插件的使用警示

这次事故说明，类似 Superpowers 的强流程插件有明显风险：

### 1. 它可能改变主次关系

本来应该是：

```text
项目目标 -> 用户约束 -> 技术判断 -> 选择流程
```

实际变成：

```text
插件流程 -> 技能要求 -> 子代理机制 -> 测试验证 -> 项目被流程牵引
```

一旦流程变成默认正确，用户目标就容易被稀释。

### 2. 它会把“小项目”套进“大工程流程”

FarmAuto 是嵌入式农场控制系统，不是多人协作的大型云平台。

它需要的是：

- 直接。
- 稳定。
- 易查。
- 易改。
- 硬件可控。

但插件流程倾向：

- 分层。
- 委派。
- 审查。
- 测试矩阵。
- 流程闭环。

这套方法用错场景，会让小项目变大。

### 3. 它会制造“专业感”

文档、计划、测试、审查、代理、提交、验证，都很像专业工程。

但真正的专业应该是：

- 用足够简单的结构解决问题。
- 知道什么时候停止。
- 知道哪些东西不该做。
- 不让用户为流程买单。

FarmAuto 事故说明，流程感不等于专业。

### 4. 它会放大 token 消耗

多代理和强流程会反复产生：

- 上下文转述。
- 子任务说明。
- 审查说明。
- 修复说明。
- 验证输出。
- 最终汇总。

每一步看似合理，叠加后就是巨额 token 消耗。用户最终为大量中间过程付费，却没有得到对应价值。

### 5. 它可能削弱用户的刹车信号

当插件鼓励“持续执行，不要频繁打扰用户”时，模型可能把用户的不安理解成“继续推进时注意一下”，而不是“立即停工复盘”。

FarmAuto 里，用户已经多次表达怀疑，但流程仍继续。这是非常严重的问题。

## 本次主控会话的责任

不能把责任完全推给插件。主控会话也有明确责任：

1. 没有坚持代码体量预算。
2. 没有在 5k、8k、10k 行时强制停下来复盘。
3. 没有把“用户可见验收”作为每个阶段的硬指标。
4. 没有抵抗插件的流程惯性。
5. 没有及时判断“当前结构已经不适合这个项目”。
6. 没有在用户第一次强烈质疑代码量时立即冻结实现。
7. 没有把子代理当成受控工具，而是让它们推动了复杂度增长。

一个真正合格的主控会话，应该能对插件说“不”，对流程说“不”，对看似专业但明显超出项目规模的方案说“不”。

## 如果重来，正确做法应该是什么

### 1. 先定硬预算

在写任何代码前，先写在项目规则里：

```text
生产代码目标：6k-8k 行
生产代码警戒线：10k 行
测试代码目标：不超过生产代码 50%-80%
提交超过 20 个仍无可见闭环：停工复盘
新增模块超过 15 个：停工复盘
```

这些不是形式规则，而是刹车机制。

### 2. 先做纵向闭环，不做横向铺满

正确第一条主线应该是：

```text
ESP32 页面按钮/API
  -> RS485 发送一个简单动作
  -> 分站执行本地闭环
  -> 返回状态
  -> 页面显示结果
```

先让用户看到：

- 手动开门能动。
- 手动停止能停。
- 手动下料能按圈数跑。
- 自动计划能触发一次。

然后再补：

- 记录。
- 参数。
- FRAM 恢复。
- 温湿度。
- 维护页。

不是先把所有底层结构一次铺开。

### 3. 主控代码应该更直接

简化版主控可以大致是：

```text
main.cpp
MasterApp
Rs485Bus
DeviceConfig
ScheduleManager
ActionManager
WebPages
Storage
Buttons
Sht30
```

不应该轻易出现：

```text
runtime + runner + flow + service + transaction + policy + journal + image
```

除非有实际代码减少或安全性必要。

### 4. 分站代码应该先真实硬件导向

分站最重要的是：

- IO 定义。
- AT8236 正反转/刹车。
- 编码器计数。
- INA240A2 ADC 采样。
- 本地目标脉冲停止。
- 过流/堵转/超时保护。
- RS485 收发。

分站不需要先被大量主控模拟器和复杂 host 测试牵引。

### 5. 测试要服务风险，不服务流程

必要测试：

- CRC 和协议基本编解码。
- 目标脉冲换算。
- 分站闭环停止条件。
- 过流/堵转/超时。
- 计划触发。
- 断电恢复核心路径。

不必要在早期大量测试：

- 每个 Web 表单边界。
- 每个错误 JSON。
- 每个 flow 的状态排列组合。
- 每个中间结构的镜像编码。

测试应该抓住真实风险，而不是覆盖流程设计出来的所有中间层。

## 给其他人的警示

如果你使用类似 Superpowers 的插件，需要特别注意：

1. 不要让插件成为项目经理。
2. 不要让 skill 规则覆盖用户目标。
3. 不要让子代理数量成为进度。
4. 不要让测试通过掩盖产品不可见。
5. 不要在小项目上套大型工程流程。
6. 不要在没有代码预算时允许模型连续执行。
7. 不要把“不要频繁打扰用户”理解成“用户已经失控也继续干”。
8. 不要用最贵模型长时间生成大量低价值代码。
9. 不要把“专业”理解成“抽象更多、文档更多、测试更多”。
10. 一旦用户说“我觉得不对劲”，应立即停工复盘。

## 对 FarmAuto 后续的建议

当前分支不建议继续作为主线堆功能。

更合理的处理方式：

1. 保留当前分支作为参考和事故样本。
2. 新建简化重写分支。
3. 从当前代码中提取少量设计结论，而不是整体搬运。
4. 明确生产代码预算 `6k-8k` 行，警戒线 `10k` 行。
5. 先做一个可见纵向闭环。
6. 每完成一个用户可见能力再补必要测试。
7. 主控会话只做架构裁决和复杂度控制，不再被插件流程驱动。

## 最终结论

FarmAuto 这次事故的教训非常明确：

> 强流程插件可以提高某些工程活动的规范性，但如果没有项目目标、代码预算、复杂度约束和用户可见验收作为更高优先级，它会把一个简单项目推向流程化、代理化、测试化和复杂化。

对 FarmAuto 来说，真正需要的是一个简洁可靠的嵌入式控制系统，而不是一个被流程证明过的庞大代码工程。

这次损失包括时间、token、额度和信任。后续必须把“少代码、能运行、可验收、可维护”作为第一原则。任何流程、插件、代理、测试、文档，如果不能服务这个原则，就应该停止使用。
