# Web 静态原型专业评审

## 评审原则

本评审按无人值守设备、远程维护、低误操作、移动端可用和后续固件可实现性判断。静态原型不是视觉稿本身，而是确认首版页面信息架构、操作入口和状态表达。

## 总体结论

当前原型方向基本正确，但还不能直接作为最终编码依据。它已经把两个应用拆成独立页面，并区分了业务页面和 Esp32Base 系统页面；自动门、喂食器的核心业务路径也基本覆盖。

主要不足不在“缺页面”，而在“页面信息优先级和交互闭环”：有些按钮是动作还是跳转不够明确，有些诊断和维护操作需要更清楚地区分，喂食器计划编辑刚拆出独立页面但还需要继续细化。

## Esp32FarmDoor

首页当前合理点：

- 首屏把门状态、当前位置、位置可信、保护状态和开关门操作放在一起，符合远程查看和快速操作。
- 不把保护最大圈数、最大时长放首页黄金位置是正确的，它们属于低频维护参数。
- 运行曲线作为自动门保留项合理，能帮助远程判断过载、卡顿、异常停止。

仍需改进：

- 首页应考虑故障场景样式：例如“位置不可信”“保护停止”“正在运行”时按钮状态如何变化。
- 运行曲线需要确认首版是否展示最近一次动作，还是只在运行中展示实时曲线。
- 维护页的“标定当前位置为开门状态/关门状态”属于危险操作，最终需要二次确认流程原型。

## Esp32FarmFeeder

首页当前合理点：

- 正在下料进度整合到通道行，比单独占一个大卡片更适合多路设备。
- 断电中断在顶部告警展示是必要的，符合“不自动续喂”的安全原则。
- 每路故障不阻塞其他路的表达基本清楚。

仍需改进：

- “启动全部可用通道”需要在页面上表达它只启动空闲、启用、无故障通道。
- 通道行里的“重新手动投喂”应进入明确的单路启动确认流程，避免误点。
- 计划编辑页已经拆出，但还需要确认新增计划、编辑已有计划、删除计划的二次确认和错误提示。

## 记录页

当前把筛选、表格、分页合并到一个记录卡片是合理的。记录页不应做成“统计页”，应保留原始业务记录分页查看。

仍需改进：

- 后续应增加时间范围筛选，否则多年记录只靠页码查找效率低。
- 导出不是首版必须，但页面上不应给用户造成“只能靠翻页查多年数据”的印象；可以在后续增强中保留导出入口。

## 诊断页

当前把硬件、存储、业务诊断集中在诊断页是合理的。

需要注意：

- “设备检查与校准”不是普通工具区，里面包含危险操作。最终页面应把普通检查、校准、格式化分层显示。
- 格式化业务存储必须明确二次确认和数据后果，不应和普通检查按钮放在同等视觉权重。
- Esp32Base 系统日志和业务诊断边界需要继续保持：业务页只给入口，不复制系统日志功能。

## 当前剩余评审项

- 自动门首页：补齐运行中、故障、位置不可信三种状态的页面效果。
- 自动门维护：补二次确认交互示例。
- 喂食器首页：补单路手动投喂确认流程。
- 喂食器计划：补删除计划、保存失败、未标定通道的错误状态。
- 诊断页：把检查、校准、危险格式化分层。
- 记录页：增加时间范围筛选是否作为首版必要项需要确认。
