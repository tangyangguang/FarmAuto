# Esp32FarmFeeder 三路喂食器重写方案

## 项目目标

Esp32FarmFeeder 是独立于 Esp32FarmDoor 的三路喂食器控制器。它不复用自动门业务逻辑，只复用公共硬件能力和 Esp32Base 接入方式。

首版提供本地 Web 控制、配置、状态查看、今日累计、每日定时投喂和长期原始记录。远程访问由网络层解决，不接入 Blinker、MQTT 或云端控制协议。

## 硬件记录

- 主控：ESP32。
- 电机驱动：AT8236 或单向 PWM 驱动方式，按实际 PCB 确认。
- 持久化存储：AT24C128。
- 喂食电机数量：3 路。
- 电机类型：带 AB 相编码器的直流减速电机。
- 当前默认电机：23rpm 减速电机。
- 当前默认减速比：270:1。
- 编码器电机轴每圈信号：A/B 相各 16 个脉冲。
- 默认输出轴每圈信号数：270 * 16 = 4320。
- 标定下料量：约 70g/输出轴圈。

默认引脚：

- 喂食器 1 PWM：GPIO16。
- 喂食器 2 PWM：GPIO17。
- 喂食器 3 PWM：GPIO18。
- 预留 PWM：GPIO19。
- 喂食器 1 编码器 A/B：GPIO33 / GPIO32。
- 喂食器 2 编码器 A/B：GPIO26 / GPIO25。
- 喂食器 3 编码器 A/B：GPIO14 / GPIO27。
- I2C SDA/SCL：GPIO21 / GPIO22。

首版暂不把物理按钮作为必需功能；如后续需要，再定义现场启动/停止按钮方案。

## 功能范围

首版功能：

- 单独启动喂食器 1、2、3。
- 单独停止喂食器 1、2、3。
- 顺序启动全部喂食器，启动间隔可配置，启动后允许多路并行运行。
- 停止全部喂食器。
- 每天定时投喂。
- 支持跳过今日定时投喂，且不删除长期计划。
- 配置每路单次投喂目标。
- 配置电机速度、软启动时长、软停止时长。
- 统计每路当天累计工作信号数。
- 显示当天累计圈数和估算下料克数。
- 日期变化时归档当天数据，并清零当天计数。
- 原始记录尽量完整保存投喂、停止、故障、跳过今日、标定和配置变化等事件，保留时间按多年目标设计。
- Web 二次确认后清空当天计数。
- 电流保护：当前硬件没有电流检测，首版只在结构上预留；未来如增加电流检测，按每个电机对应一个 INA240 芯片设计。

明确不做：

- 不接入 Blinker App。
- 不实现 MQTT 或 HTTP 云端控制。
- 不迁移老项目 AT24C128 数据。
- 不把喂食器逻辑和自动门逻辑写成同一个固件。
- 不修改 Esp32Base。
- 不修改 old_prj。
- 当前阶段不写源码、不创建 PlatformIO 工程结构。

## 用户配置模型

用户主要配置项建议使用克数，而不是圈数：

```text
目标克数 -> 目标圈数 -> 目标编码器信号数
```

每路应支持独立标定：

- 单次目标克数。
- 每圈下料克数。
- 减速比。
- 编码器电机轴每圈脉冲数。
- 输出轴每圈信号数。

显示层可以同时显示克数、圈数和脉冲数。持久化应优先保存原始信号数，避免多次换算造成精度损失。

## 控制模型

Esp32FarmFeeder 应由应用层 `FeederController` 组合公共库：

```text
FeederController
  3 x Esp32EncodedDcMotor
  Esp32MotorCurrentGuard
  Esp32At24cRecordStore
  FeedingHistory
```

`FeedingHistory` 属于 Esp32FarmFeeder 应用模块，不进入公共库。

每路电机状态建议：

```text
Idle / SoftStarting / Running / SoftStopping / Fault
```

启动全部时，应按配置间隔顺序启动，降低启动电流叠加；已启动通道可以继续运行。单路故障时，该通道停止并记录故障，其他通道继续运行。停止全部的策略仍需在实现前确认。

## 定时投喂方案

首版定时投喂只支持“每天执行”。

需要支持：

- 每日固定时间执行一次投喂计划。
- 支持启用/禁用每日计划。
- 支持跳过今日；跳过只影响当天，不删除长期计划。
- 跳过今日必须持久化，防止重启后误投喂。
- 定时触发后按顺序启动全部启用通道，并使用 `startAllIntervalMs` 间隔。
- 单通道故障时其他通道继续，最终状态记录为部分成功。

待后续确认：

- 日期/时间来源失败时是否允许手动投喂。
- 日期/时间来源失败时是否暂停自动定时投喂。
- 当天错过计划时间后是否补投喂。

## 电流保护问题

Esp32FarmFeeder 当前硬件没有电流检测芯片，因此首版不能依赖电流保护判断堵转。

首版应依靠以下保护：

- 编码器无脉冲堵转检测。
- 最大运行时间。
- 最大运行圈数或最大运行脉冲。
- 启动全部时的顺序启动间隔。

未来如果增加电流检测，推荐硬件方案是每个电机对应一个 INA240 芯片，也就是三路电机对应三路独立电流检测。这样每路可以独立判断过流、堵转或负载异常。

不推荐三路共用一个总电流检测通道。总电流只能判断系统级异常，难以可靠定位具体哪一路电机异常。

## 持久化数据

配置数据：

- 3 路单次投喂目标克数。
- 3 路每圈下料克数。
- 电机速度。
- 软启动时长。
- 软停止时长。
- 减速比。
- 编码器电机轴每圈脉冲数。
- 输出轴每圈信号数。
- 加速/减速和堵转检测参数。
- 最大运行时间。
- 最大运行圈数。
- 电流保护配置。

运行数据：

- 当前日期。
- 3 路当天累计信号数。
- 定时计划和跳过今日状态。
- 长期原始记录索引或摘要。

原始记录应尽量完整保存，显示时再换算成圈数和克数。AT24C128 不适合作为多年原始记录主存储；它应优先保存关键配置、状态、计划、今日计数和必要索引。多年原始记录需要单独评估 ESP32 flash 文件系统、外部 FRAM、SD 卡或其他介质，并明确磨损均衡、导出和容量告警策略。

## 待确认与实验项

- 3 路电机是否完全相同。
- 每路实际下料量是否都约 70g/圈。
- 首版是否以克数作为主要配置项。
- 是否需要物理按钮。
- 未来电流检测是否采用每电机一个 INA240 芯片。
- 停止全部的默认策略。
- 每日定时投喂的默认时间。
- 错过定时投喂后是否补投喂。
- 多年原始记录的存储介质和导出策略。
