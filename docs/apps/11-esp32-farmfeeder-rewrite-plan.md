# Esp32FarmFeeder 三路喂食器重写方案

## 项目目标

Esp32FarmFeeder 是独立于 Esp32FarmDoor 的三路喂食器控制器。它不复用自动门业务逻辑，只复用公共硬件能力和 Esp32Base 接入方式。

首版提供本地 Web 控制、配置、状态查看、今日累计、每日定时投喂和长期原始记录。远程访问由网络层解决，不接入 Blinker、MQTT 或云端控制协议。

## 硬件记录

- 主控：ESP32。
- 电机驱动：AT8236 或单向 PWM 驱动方式，按实际 PCB 确认。
- 持久化存储：AT24C128。
- 喂食电机数量：3 路。
- 电机类型：带 AB 相编码器的直流减速电机。
- 当前默认电机：23rpm 减速电机。
- 当前默认减速比：270:1。
- 编码器电机轴每圈信号：A/B 相各 16 个脉冲。
- 默认输出轴每圈信号数：270 * 16 = 4320。
- 当前标定下料量：约 70g/输出轴圈，但以后可能变化。
- 三路电机属于同系列，但转速比可能不一样，必须支持每路独立参数。
- 每路都有独立饲料桶和下料器，饲料桶容量可能不同，必须支持每路独立配置桶容量和余量估算。

默认引脚：

- 喂食器 1 PWM：GPIO16。
- 喂食器 2 PWM：GPIO17。
- 喂食器 3 PWM：GPIO18。
- 预留 PWM：GPIO19。
- 喂食器 1 编码器 A/B：GPIO33 / GPIO32。
- 喂食器 2 编码器 A/B：GPIO26 / GPIO25。
- 喂食器 3 编码器 A/B：GPIO14 / GPIO27。
- I2C SDA/SCL：GPIO21 / GPIO22。

首版不需要物理按钮。所有控制优先通过本地 Web/API 完成。

## 功能范围

首版功能：

- 单独启动喂食器 1、2、3。
- 单独停止喂食器 1、2、3。
- 顺序启动全部喂食器，启动间隔可配置，启动后允许多路并行运行。
- 停止全部喂食器。
- 每天定时投喂。
- 支持跳过今日定时投喂，且不删除长期计划。
- 配置每路单次投喂目标。
- 配置每路饲料桶容量和当前余量。
- 饲料桶余量第一版采用软件扣减估算；硬件感知作为下一阶段增强。
- 配置电机速度、软启动时长、软停止时长。
- 统计每路当天累计工作信号数。
- 显示当天累计圈数和估算下料克数。
- 日期变化时归档当天数据，并清零当天计数。
- 原始记录尽量完整保存投喂、停止、故障、跳过今日、标定和配置变化等事件，保留时间按多年目标设计。
- Web 二次确认后清空当天计数。
- 电流保护：当前硬件没有电流检测，首版只在结构上预留；未来如增加电流检测，按每个电机对应一个 INA240A2 芯片设计。

明确不做：

- 不接入 Blinker App。
- 不实现 MQTT 或 HTTP 云端控制。
- 不迁移老项目 AT24C128 数据。
- 不把喂食器逻辑和自动门逻辑写成同一个固件。
- 不修改 Esp32Base。
- 不修改 old_prj。
- 当前阶段不写源码、不创建 PlatformIO 工程结构。

## 用户配置模型

首版不强制只使用克数或只使用圈数。推荐使用“目标模式”配置：

```text
targetMode = Grams | Revolutions
```

原因：

- 克数更贴近实际投喂目标，但依赖每路下料标定和饲料状态。
- 圈数更接近机械控制，适合标定前、调试期或饲料密度变化较大的场景。
- 公共控制最终都应换算成目标编码器信号数执行。

每路应支持独立标定：

- 单次目标模式。
- 单次目标克数。
- 单次目标圈数。
- 每圈下料克数。
- 减速比。
- 编码器电机轴每圈脉冲数。
- 输出轴每圈信号数。
- 饲料桶容量。
- 当前估算余量。

饲料桶余量方案见 `docs/apps/20-feeder-bucket-level-sensing.md`。第一版不加余量传感器，补料时通过 Web 设置当前余量或记录加料量；每次投喂完成后按实际运行圈数和每圈克数扣减。下一阶段如果需要硬件感知，优先考虑每路料桶独立称重。

显示层可以同时显示克数、圈数和脉冲数。持久化应优先保存原始信号数，避免多次换算造成精度损失。

## 控制模型

Esp32FarmFeeder 应由应用层 `FeederController` 组合公共库：

```text
FeederController
  3 x Esp32EncodedDcMotor
  future 3 x Esp32MotorCurrentGuard
  Esp32At24cRecordStore
  FeederRecordService
```

`FeederRecordService` 属于 Esp32FarmFeeder 应用模块，不进入公共库。

每路电机状态建议：

```text
Idle / SoftStarting / Running / SoftStopping / Fault
```

启动全部时，应按配置间隔顺序启动，降低启动电流叠加；已启动通道可以继续运行。单路故障时，该通道停止并记录故障，其他通道继续运行。

停止全部默认策略：

- 普通停止全部：同时向所有运行通道发出停止请求，每路按自己的软停止策略停机。
- 故障或急停停止全部：同时向所有运行通道发出急停请求。
- 不需要像启动一样错峰停止，因为停止不会产生启动浪涌；同时停止更符合远程安全预期。

## 定时投喂方案

首版定时投喂只支持“每天执行”。

需要支持：

- 每日固定时间执行一次投喂计划。
- 支持启用/禁用每日计划。
- 支持跳过今日；跳过只影响当天，不删除长期计划。
- 跳过今日必须持久化，防止重启后误投喂。
- 定时触发后按顺序启动全部启用通道，并使用 `startAllIntervalMs` 间隔。
- 单通道故障时其他通道继续，最终状态记录为部分成功。
- 未配置每日时间时，不自动定时投喂。
- 日期/时间来源失败时，暂停自动定时投喂。
- 日期/时间来源失败不影响手动投喂。
- 手动和定时按通道独立仲裁：某一路正在自动喂食时，该路手动启动返回 `Busy`，其他空闲可用通道仍可手动启动。
- 定时计划触发时，已运行通道不排队、不补执行，记录 busy skipped；其他空闲可用计划通道继续启动。
- 当天错过计划时间后不补投喂，只记录 missed 事件。

## 电流保护问题

Esp32FarmFeeder 当前硬件没有电流检测芯片，因此首版不能依赖电流保护判断堵转。

首版应依靠以下保护：

- 编码器无脉冲堵转检测。
- 最大运行时间。
- 最大运行圈数或最大运行脉冲。
- 启动全部时的顺序启动间隔。

未来如果增加电流检测，推荐硬件方案是每个电机对应一个 INA240A2 芯片，也就是三路电机对应三路独立电流检测。这样每路可以独立判断过流、堵转或负载异常。

不推荐三路共用一个总电流检测通道。总电流只能判断系统级异常，难以可靠定位具体哪一路电机异常。

更好的方案不是共用一个传感器，而是在 PCB 布局、采样电阻、ADC 引脚和接地布线上为每路 INA240A2 预留一致的测量条件。这样软件和诊断都更简单。

## 参数与持久化数据

系统配置数据优先注册到 Esp32Base App Config：

- 3 路电机速度。
- 3 路软启动时长。
- 3 路软停止时长。
- 3 路减速比。
- 编码器电机轴每圈脉冲数。
- 编码器计数模式。
- 3 路输出轴每圈信号数。
- 加速/减速和堵转检测参数。
- 最大运行时间。
- 最大运行圈数。
- 未来电流保护硬件和阈值配置。

业务页面管理的数据：

- 3 路单次投喂目标模式、目标克数和目标圈数。
- 每日定时计划和跳过今日。
- 3 路饲料桶容量。
- 3 路低余量告警阈值。
- 3 路每圈下料克数标定。
- 3 路饲料桶当前估算余量。

运行数据：

- 当前日期。
- 3 路当天累计信号数。
- 定时计划和跳过今日状态。
- 长期原始记录索引或摘要。

原始记录应尽量完整保存，显示时再换算成圈数和克数。AT24C128 不适合作为多年原始记录主存储；它应优先保存关键恢复状态、计划执行状态、今日计数和必要索引。普通系统参数配置优先使用 Esp32Base App Config。多年原始记录首版优先考虑 ESP32 flash 文件系统，并明确磨损均衡、导出、轮转和容量告警策略。

关键数据记录应尽量全面但不过度设计：记录有诊断和统计价值的事件、原始计数和结果，不记录每个编码器脉冲或无意义高频采样。

## 待确认与实验项

- 每路电机的实际减速比和 `outputPulsesPerRev`。
- 每路实际下料量标定值；当前约 70g/圈，但以后可能变化。
- 克数模式和圈数模式的 Web 表单交互。
- 每路饲料桶容量单位、默认值和余量校准方式。
- ESP32 flash 文件系统实际分区容量；推荐 1MB 起步、空间允许时 2MB。
