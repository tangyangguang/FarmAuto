# Web 页面工作流细化

## 目标

本文细化 Esp32FarmDoor 和 Esp32FarmFeeder 的页面工作流、状态字段、按钮启用条件和危险操作确认规则。它补充 `19-web-page-prototypes.md`，不替代应用 API 文档。自动门 API 见 `24-esp32-farmdoor-web-api.md`，喂食器 API 见 `25-esp32-farmfeeder-web-api.md`。

Esp32Base 系统页面、系统日志、App Config 与应用业务页面/API 的边界见 `23-esp32base-web-integration.md`。

原则：

- 页面服务于远程操作和无人值守维护，不做展示型页面。
- 所有长动作都只发起命令，页面通过 `commandId` 或状态轮询显示进度。
- 所有危险操作必须二次确认并写入长期事件。
- 页面显示的业务状态必须能追溯到 API snapshot、事件记录或持久化记录。

## 通用页面行为

状态刷新：

- 首页运行中默认 1s 刷新一次。
- 空闲时默认 5s 到 10s 刷新一次。
- 维护流程执行中默认 1s 刷新一次。
- 记录页和导出页不自动高频刷新。

按钮规则：

- 设备处于 `Booting` 时禁用所有动作按钮。
- 任意电机运行时，只保留当前动作相关按钮和 `停止`。
- `停止` 在运行中始终可见。
- `格式化存储`、`设置当前位置`、`保存端点`、`清空今日计数`、`标定每圈下料量` 必须二次确认。

错误展示：

- `Busy`：显示当前运行中的 commandId 或业务状态。
- `PositionUnknown`：隐藏普通开关门按钮，显示维护入口。
- `StorageError`：显示 AT24C 和 flash 诊断入口。
- `OnsiteRequired`：显示原因和最近诊断，不继续提供远程危险操作。

## Esp32FarmDoor 首页

必须显示：

- 门状态：`PositionUnknown` / `IdleClosed` / `IdleOpen` / `IdlePartial` / `Opening` / `Closing` / `Fault`。
- 当前位置：pulses、百分比、位置来源、最近保存时间。
- 端点：关闭点、开门目标、安全上限。
- 电机状态：公共电机库 state、速度、输出百分比、剩余脉冲。
- 电流状态：当前 mA、滤波 mA、阈值、保护状态。
- 限位状态：第一版显示 `未安装/禁用`；下一阶段显示触发和断线状态。
- 存储状态：AT24C 在线、flash 余量、最近写入错误。

按钮启用：

| 状态 | 开门 | 关门 | 停止 | 清除故障 | 维护入口 |
| --- | --- | --- | --- | --- | --- |
| `PositionUnknown` | 禁用 | 禁用 | 禁用 | 禁用 | 启用 |
| `IdleClosed` | 启用 | 禁用 | 禁用 | 禁用 | 启用 |
| `IdleOpen` | 禁用 | 启用 | 禁用 | 禁用 | 启用 |
| `IdlePartial` | 启用 | 启用 | 禁用 | 禁用 | 启用 |
| `Opening` / `Closing` | 禁用 | 禁用 | 启用 | 禁用 | 只读 |
| `Fault` | 禁用 | 禁用 | 视情况 | 启用 | 启用 |

普通开门/关门：

- API 只发起命令，不等待完成。
- 返回后页面显示 commandId、目标位置、预计最大时间。
- 运行曲线显示位置、速度、电流、故障点。

## Esp32FarmDoor 维护流程

### 无限位端点维护

适用：第一版未安装开门/上限位，或位置不可信。

流程：

1. 用户进入维护页，页面显示“无限位维护风险提示”。
2. 用户点击“进入端点维护”，后端进入 `Maintenance`。
3. 页面只显示低速点动、停止、设置关闭点、保存开门目标、验证端点。
4. 用户低速点动到关闭位置，点击“设置当前位置为关闭点”。
5. 用户低速点动到打开位置，点击“保存当前位置为开门目标”。
6. 系统生成 `maxRunPulses` 和 `maxCloseUnwindPulses` 的推荐值。
7. 用户执行“低速验证端点”。
8. 验证成功后退出 `PositionUnknown`，允许普通控制。

点动限制：

- 默认速度 30%。
- 单次点动默认最多 1000ms。
- 如果已有开门目标，单次点动最多 5% 开门目标脉冲。
- 每次点动都必须记录方向、时长、脉冲变化和停止原因。

失败处理：

- 点动无编码器脉冲：立即停止，进入故障或保持维护状态。
- 电流超限：立即停止，记录电流保护事件。
- 保存端点失败：不允许退出维护流程。
- 低速验证失败：保持 `PositionUnknown` 或 `Fault`。

### INA240A2 零点校准

前置条件：

- 电机停止。
- 用户确认当前无电机电流。
- ADC 状态正常。

保存内容：

- 零点 mV。
- ADC 原始值。
- 校准时间。
- 采样数量。
- 最大/最小波动。

## Esp32FarmDoor 诊断包

诊断包应包含：

- 当前 status snapshot。
- 系统配置摘要，来源优先为 Esp32Base App Config。
- 当前位置和端点快照。
- 最近 100 条事件。
- AT24C inspect。
- flash 容量和长期记录范围。
- 最近一次电流 trace 摘要。
- Esp32Base 系统日志入口 `/esp32base/logs` 的链接和最近系统日志摘要可选，不复制为业务记录。

## Esp32FarmFeeder 首页

必须显示：

- 全局状态：`Idle` / `Starting` / `Running` / `Stopping` / `RollingDay` / `Degraded` / `Fault` / `Maintenance`。
- 今日计划：启用状态、计划时间、今日是否已执行、是否跳过今日。
- 通道状态表：每路启用状态、运行状态、目标模式、目标量、今日累计、估算余量、故障摘要。
- 运行通道数、启用通道数、故障通道数。
- 存储状态：AT24C、flash、记录范围。

按钮启用：

- 某一路运行时，只禁用该路新的启动按钮；其他空闲、已启用、已安装且无故障通道仍可手动启动。
- 启动全部在有通道运行时仍可用，但只会启动空闲可用通道，并在结果中显示 busy/skipped 通道。
- `停止全部` 在任一路运行时启用。
- 单路故障不禁用其他正常通道；页面显示 `Degraded` 告警或运行中降级提示。
- `跳过今日` 只影响定时投喂，不影响手动投喂。

## Esp32FarmFeeder 投喂计划页

每路目标：

- 使用 segmented control 选择 `克数` 或 `圈数`。
- 切换模式不删除另一模式已保存值。
- 页面同时显示换算后的圈数、脉冲和估算克数。
- 未完成每圈克数标定时，克数模式应显示 `NotConfigured`，但圈数模式可用。

每日计划：

- 未设置时间时，计划自动禁用。
- 错过计划时间不补投喂，只记录 missed 事件。
- 时间无效时暂停自动定时，但手动投喂仍可用。

## Esp32FarmFeeder 饲料桶页

每路显示：

- 容量。
- 当前估算余量。
- 余量百分比。
- 低余量阈值和严重低余量阈值。
- 预计可投喂次数。
- 预计剩余天数。
- 最近一次补料记录。

维护动作：

- `设置当前余量`：直接设置估算余量，二次确认。
- `记录加料量`：当前余量增加指定克数，不超过容量。
- `标记已加满`：当前余量设置为容量。
- `修正容量`：改变容量时，不自动改变当前余量，除非用户选择同步填满。

记录要求：

- 每次补料动作记录操作类型、原余量、新余量、变化量。
- 每次投喂扣减记录原余量、新余量、本次估算消耗。
- 估算余量为 0 但继续投喂时记录 `BucketEstimateUnderflow`。

## Esp32FarmFeeder 标定流程

### 每圈下料克数标定

流程：

1. 用户选择通道。
2. 页面提示准备容器接料。
3. 用户发起固定圈数测试，默认 1 圈或较小安全值。
4. 电机完成后，页面显示实际脉冲、圈数和运行时间。
5. 用户输入实测重量。
6. 系统计算 `gramsPerRevolution` 并保存校准记录。

限制：

- 标定运行必须限制最大时间和最大脉冲。
- 标定期间不允许其他通道启动。
- 标定失败不更新旧校准值。

### 单路小剂量测试

用途：

- 验证方向。
- 验证编码器计数。
- 验证下料是否通畅。

默认：

- 使用圈数模式。
- 目标小于正常投喂量。
- 不计入今日正式投喂量，但写入维护事件。

## 记录页

查询条件：

- 时间范围。
- 事件类型。
- 通道编号。
- 故障原因。

分页规则：

- 默认按时间倒序。
- 每页返回有限条数。
- 大范围导出使用异步命令或分块下载。

导出：

- JSON Lines：保留结构化字段，适合后续程序分析。
- CSV：适合人工查看和表格分析。

## 源码前检查

- Esp32Base App Config 字段容量是否足够。
- Esp32Base route 数量是否足够。
- 应用页面是否避开 `/logs` 和 `/api/logs`，业务记录是否统一使用 `/records` 和 `/api/app/records`。
- 文件导出是否需要 Esp32Base 增强流式下载能力。
- 手机窄屏下首页表格是否改为每路折叠块。
- 危险操作 confirm token 是否由应用实现，还是需要 Esp32Base 提供通用能力。
