# 公共库边界与 Esp32Base 关系

## 目标

FarmAuto 的 `lib/` 用来沉淀可跨项目复用的硬件和存储能力。公共库不是具体应用项目的一部分，也不应包含开门、投喂、历史统计、页面交互等应用业务。

公共库文档应只描述通用能力、约束、接口边界和芯片支持策略。具体应用如何使用公共库，写在 `docs/apps/`。

## 当前公共库

当前规划三个公共库：

- `EncodedDcMotor`：带编码器 DC 电机运动控制。
- `MotorCurrentGuard`：电机电流采样与过流/堵转辅助保护。
- `At24cRecordStore`：AT24C/24LC/24AA 系列 I2C EEPROM 可靠记录存储。

这些库未来可能被 FarmAuto 之外的其他 Esp32 项目使用，因此命名和文档都不绑定具体应用。

## 与 Esp32Base 的分工

Esp32Base 是 ESP32 应用基础底座，负责跨设备、跨业务的系统基础能力：

- 应用生命周期。
- Profile 裁剪。
- WiFi、配网、mDNS、NTP。
- Web 基础能力。
- OTA。
- 文件系统。
- 运行时日志。
- I2C/SPI 等总线基础入口。
- Watchdog、健康状态、诊断。

FarmAuto 公共库负责具体硬件或设备部件能力：

- 某类电机控制。
- 某类电流检测和保护。
- 某类外部 EEPROM 上的记录存储。

应用项目负责业务编排：

- 自动门状态机。
- 喂食器三路编排。
- Web 页面和 API。
- 用户配置语义。
- 业务事件和历史统计。

## 依赖原则

公共库可以分三层考虑依赖：

```text
纯 Arduino/ESP32 能力
  -> 可在多个 Esp32 项目中复用

可选 Esp32Base 适配
  -> 使用 Esp32Base 的日志、总线或诊断能力，但核心逻辑不强绑定

应用项目组合层
  -> 具体 Esp32 应用项目
```

首选设计是：公共库核心逻辑尽量不强依赖 Esp32Base，只依赖 Arduino ESP32 和必要第三方库。这样其他 Esp32 项目也能复用。

如果某个公共库需要 Esp32Base 的能力，应优先通过很薄的适配层接入。例如：

- 日志输出可以接受回调，而不是直接写死 Esp32Base 日志。
- I2C 总线可以接受 `TwoWire&` 或总线接口，而不是在库内独占初始化。
- 诊断信息可以返回结构化状态，由应用决定如何上报。

## 什么时候留在 FarmAuto 公共库

满足以下条件时，能力应留在 FarmAuto `lib/`：

- 面向具体硬件芯片、传感器、执行器或存储器。
- 不是所有 Esp32Base 用户都会需要。
- API 仍在随实际设备演进。
- 主要服务 FarmAuto 和相近设备项目。
- 放进 Esp32Base 会增加基础库体积或概念负担。

例如：

- `EncodedDcMotor`。
- `MotorCurrentGuard`。
- `At24cRecordStore`。

## 什么时候沉淀到 Esp32Base

只有证据充分时，才考虑把能力沉淀到 Esp32Base。触发条件包括：

- 多个无关项目都需要同一基础能力。
- 能力明显属于 ESP32 系统底座，而不是某个设备部件。
- 当前 Esp32Base 能力缺口已经阻塞应用项目。
- 发现 Esp32Base 可复现 bug。
- API 已经稳定，放入基础库不会引入明显体积和维护负担。

可能适合 Esp32Base 的例子：

- 通用非阻塞任务调度辅助。
- 通用诊断状态模型。
- 通用 Web API 注册机制缺口。
- 总线管理的基础能力缺口。

不应轻易放入 Esp32Base 的例子：

- 某个电机驱动。
- 某个电流传感器后端。
- 某个 EEPROM 记录格式。
- 某个应用的 Web 页面。

## 什么时候独立成单独仓库

如果某个公共库开始被多个仓库使用，并且已经具备稳定 API，可以考虑从 FarmAuto `lib/` 拆成独立 PlatformIO/Arduino 库。

拆分条件：

- 至少两个项目真实使用。
- 有清晰 README、示例和测试。
- 不依赖 FarmAuto 应用代码。
- 版本边界清晰。
- API 变更策略清楚。

在此之前，先留在 FarmAuto 内部 `lib/`，避免过早增加仓库和版本管理成本。

## 芯片支持策略

公共库可以支持多个芯片型号，但必须避免为了未来可能性提前做复杂框架。

推荐策略：

- 当前真实使用的芯片优先实现。
- 同类芯片用配置扩展，不为每个型号复制一套逻辑。
- 读取方式差异很大的芯片使用后端接口。
- 文档可以记录未来支持方向，但源码只实现当前验证过的路径。

例子：

- `At24cRecordStore` 通过容量、页大小、地址字节数等配置支持 AT24C 系列。
- `MotorCurrentGuard` 首版实现 INA240A2，未来用后端支持 ACS712 和 INA226。
- `EncodedDcMotor` 通过驱动后端支持单向 PWM 和 H 桥，不把业务动作写进库。
