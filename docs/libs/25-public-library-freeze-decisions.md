# 公共库接口冻结前推荐决策

## 目标

本文记录三个公共库进入源码骨架前的推荐选择。原则是不过度设计、接口长期可维护、适合无人值守设备和远程维护。

本文不是最终源码接口，但作为后续接口冻结评审的默认答案。除非实机测试或用户确认推翻，后续设计优先按本文执行。

## 总体原则

| 项目 | 推荐选择 | 原因 |
| --- | --- | --- |
| 依赖方向 | 公共库核心不强依赖 Esp32Base | 未来可独立为通用 Arduino/PlatformIO 库，应用层再接 Esp32Base 日志、Web、诊断 |
| 运行模型 | 全部采用非阻塞 `update(nowMs)` / `readSample()` 模式 | 避免阻塞上层循环、输入处理、看门狗、日志和多执行器调度 |
| 错误表达 | 返回明确结果码 + snapshot | 方便远程诊断，不依赖串口日志 |
| 配置方式 | `begin()` 传硬件资源，`configure()` 传运行参数 | 硬件资源和可调参数分离，便于远程维护 |
| 日志 | 公共库只暴露事件/状态，不直接写应用日志 | 日志格式和保存位置属于应用或 Esp32Base |
| 持久化 | 公共库不保存自身配置 | 保存时机和版本迁移属于应用层 |

## Esp32EncodedDcMotor

| 问题 | 推荐选择 | 简要原因 |
| --- | --- | --- |
| 首版后端 | 支持 `SinglePwmDriver` 和 `At8236HBridgeDriver` | 覆盖当前已知硬件驱动方式，复杂度可控 |
| 编码器后端 | 首版优先使用成熟 ESP32 编码器/PCNT 能力，并用适配层隔离 | 避免库接口绑定某个第三方实现，后续可替换 |
| 计数模式 | 支持 `X1` / `X2` / `X4` 配置，默认 `X1` | 当前参数按单边 16 脉冲口径记录，默认 X1 最不容易误配；需要高分辨率时可切 X4 |
| 方向反转 | 必须支持 motor direction 和 encoder direction 独立反转 | 实际接线经常反向，远程维护时不能靠重接线解决 |
| 位置类型 | 使用 `int64_t` 累计脉冲 | 长时间运行和多圈累计不容易溢出 |
| 圈数换算 | `outputPulsesPerRev` 允许由参数计算，也允许实测覆盖 | 减速比、编码器口径和机械误差可能与标称不完全一致 |
| 状态命名 | 使用 `SoftStarting` / `SoftStopping` | 软启动和软停止成对命名，状态机更一致 |
| 软启动 | 必须支持独立 `softStartMs`，默认 1000ms | 降低启动冲击和误判堵转风险 |
| 软停止 | 必须支持独立 `softStopMs`，默认 500ms | 停止更平顺，但不拖慢故障急停 |
| 加减速曲线 | 首版只做线性 | 简单、可解释、易测试；S 曲线等实机证明需要再加 |
| 最小有效速度 | 支持 `minEffectiveSpeedPercent`，默认 15% | 低速区可能无力或无脉冲，避免软启动/软停止阶段误判 |
| 启动宽限 | 默认覆盖软启动初期，建议 `max(softStartMs, 1000ms)` | 电机刚启动时脉冲少、电流高，过早判断会误报 |
| 运行中命令 | 运行中收到新运行命令默认返回 `Busy` | 避免上层意外打断运动，上层先停再发新命令 |
| 正常停止策略 | 默认 `SoftStopThenBrake` | 无人值守设备更需要停稳和位置可控 |
| 故障停止策略 | 默认 `EmergencyStop`，H 桥可配置为 Brake 或 Coast | 故障优先安全，具体刹车/滑行由机械负载实测决定 |
| API 命名 | 运动命令使用 `requestMove*`，不使用业务化找零点命名 | 明确非阻塞请求语义，不受 old_prj 或其他库命名影响 |
| 零点语义 | 不提供专门的业务化找零点 API；使用 `requestMoveToPosition(0)` 表示运行到编码器坐标 0 | 机械找零点、限位寻找和零点校准属于应用层，公共库只处理通用位置坐标 |
| 外部保护 | 提供 `requestEmergencyStop(ExternalFault)` | 外部保护条件由上层判断后传入 |
| 图表支持 | 提供 `MotorTracePoint` 数据，不生成图表 | 支持上层远程诊断，同时不耦合 Web/UI |
| 多电机编排 | 首版不做 | 多执行器启动、停止和故障策略属于上层应用，公共库保持单电机稳定 |
| PID 速度闭环 | 首版不做 | 当前需求是定量运动和保护，不需要增加调参复杂度 |

仍需实测确认：

- X1 计数下的实际 `outputPulsesPerRev`。
- 不同负载设备各自的 `minEffectiveSpeedPercent`。
- 故障时 Brake 是否会造成机械冲击。
- 不同负载下的无脉冲堵转阈值。

## Esp32MotorCurrentGuard

| 问题 | 推荐选择 | 简要原因 |
| --- | --- | --- |
| 首版芯片 | 只实现 INA240A2 | 当前真实使用，且适合 PWM 电机电流检测 |
| 库命名 | 保持 `Esp32MotorCurrentGuard`，后端名出现 `Ina240A2` | 公共库不绑定单芯片，具体后端清晰表达芯片型号 |
| ACS712 / INA226 | 只保留芯片类命名和设计方向，不写首版实现 | 读取方式和噪声模型不同，没硬件需求时不提前复杂化 |
| 分层 | 具体芯片采样类负责采样换算，`MotorCurrentGuard` 负责保护判定 | 方便测试、替换芯片和多通道复用 |
| 抽象接口 | 首版不抽正式 `CurrentSensor` 虚接口 | 只有一个真实芯片实现时，接口会增加复杂度而收益不足 |
| 采样模型 | 上层周期读取 sample，再调用 guard update | 非阻塞，采样频率由设备目标和硬件资源决定 |
| 自动停机 | 公共库不直接停止电机 | 停哪台电机、如何记录故障属于应用层 |
| 电流方向 | INA240A2 默认支持双向电流 | H 桥正反转时更自然，远程诊断信息更完整 |
| 默认启用 | 公共库不定义默认启用策略 | 是否启用取决于上层应用是否具备对应硬件 |
| 保护判定 | 同时支持 `confirmationMs` 和 `confirmationSamples` | 时间和样本双约束更稳，仍然简单 |
| 滤波 | 首版使用一阶 IIR，默认 `filterAlpha=0.2` | 比窗口更省内存，实现简单，足够做保护 |
| 启动宽限 | 默认 1000ms，应用可按电机软启动覆盖 | 避免启动瞬间电流误判 |
| 故障阈值 | 公共库不提供固定阈值 | 电机型号、采样电阻、负载和机械结构差异很大，必须由上层配置并实机校准 |
| 传感器异常 | ADC 饱和、零点非法、读取失败进入 `SensorFault` | 远程无人值守时，传感器不可信不能继续当作正常保护 |
| 多通道 | 一个传感器实例对应一个电机通道 | 多执行器设备可以按通道独立组合，异常来源更清楚 |
| 总电流通道 | 只作为系统级异常，不定位具体电机 | 避免错误归因 |
| 图表支持 | 提供 `CurrentTracePoint` 数据，不生成图表 | 支持上层远程诊断，同时不耦合 Web/UI |

仍需实测确认：

- INA240A2 供电电压、零点电压、ADC attenuation。
- Rsense 实际阻值。
- 正常正转/反转负载电流。
- 堵转或接近堵转电流。
- PWM 噪声下采样频率和滤波参数。

## Esp32At24cRecordStore

| 问题 | 推荐选择 | 简要原因 |
| --- | --- | --- |
| 首版实测型号 | AT24C128 | 当前已知硬件计划使用，优先保证真实硬件可靠 |
| 型号扩展 | 首版承诺 AT24C128，推荐兼容 2 字节地址 AT24C32/64/256/512 | 只用配置描述容量、页大小、地址字节数，不复制逻辑 |
| 小容量寻址 | 首版不支持需要特殊地址位映射的 AT24C02/04/08/16 | 本项目一般用不到，避免为低价值型号增加特殊代码 |
| 存储模型 | wear-levelled record ring + latest sequence | 比文件系统简单，断电恢复明确，同时提供静态磨损均衡 |
| 自动格式化 | 禁止自动格式化，必须显式调用 `format()` | 保护用户配置、校准和历史数据 |
| 写前比较 | 默认开启，相同内容返回 `Unchanged` | 降低 EEPROM 写入次数 |
| 记录状态 | 首版使用 `Empty` / `Writing` / `Valid`，不主动写 `Retired` | 少一次写入，降低寿命消耗，读取时按 sequence 选最新即可 |
| 写入原子性 | 先写 `Writing` 完整记录，校验后再更新为 `Valid` | 断电时可忽略未完成记录，保留上一个有效记录 |
| CRC | header CRC 排除自身字段，payload CRC 单独计算 | 容易实现和排错 |
| 字节序 | 固定小端 | ESP32 原生友好，文档明确后跨版本可维护 |
| sequence | `uint32_t` 递增，回绕用半区间比较 | 简单够用，避免回绕时误选旧记录 |
| payload 大小 | 每个 region 固定 `slotSize`，超长返回 `PayloadTooLarge` | 不做动态分配，容量预算可控 |
| 磨损均衡 | 每个 recordType 独立记录环，按 slot 轮转写入 | 简单、可靠、可远程诊断，符合小型 EEPROM 最佳实践 |
| 高频写入 | 通过更大的 `slotCount` 或减少保存频率处理 | 不用全局动态 wear leveling 换复杂度 |
| 图表支持 | 提供 `inspect()` 磨损和错误诊断数据，不生成图表 | 支持远程查看寿命、slot 分布、CRC/校验失败趋势 |
| 设备离线 | 返回 `DeviceOffline`，不伪造默认成功 | 远程维护必须能看到存储硬件异常 |
| 既有私有格式迁移 | 首版不做 | 公共库不背特定历史格式包袱 |
| 坏块管理 | 首版不做复杂坏块管理 | EEPROM 容量小，先靠多槽和写前比较；实测需要再加 |
| FRAM 支持 | 首版不纳入 AT24C 库承诺范围 | FRAM 特性不同，未来优先做独立库，验证重复后再抽通用记录层 |

仍需实测确认：

- AT24C128 页大小、写周期和 ACK polling 表现。
- 目标 PCB 上的 I2C 地址。
- 已知设备的 region 容量是否还有升级余量。
- 高频 recordType 的预计 writesPerDay 和 slotCount。
- 断电测试下 `Writing -> Valid` 流程是否满足预期。

## 暂不进入公共库的能力

| 能力 | 推荐位置 | 原因 |
| --- | --- | --- |
| 机械端点/限位语义 | 应用层 | 外部传感器代表什么业务状态，不是通用电机概念 |
| 多执行器编排 | 应用层 | 多路启动、停止、故障策略是业务规则 |
| Web/API | 应用层，使用 Esp32Base 能力 | 页面和路由强业务相关 |
| 配置默认值 | 应用层 | 不同设备默认值不同 |
| 历史统计 | 应用层 | 统计口径属于业务，不是存储库职责 |
| 远程维护流程 | 应用层 | 需要结合状态机、Web、日志和具体硬件 |
