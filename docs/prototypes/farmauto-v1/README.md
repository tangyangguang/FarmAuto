# FarmAuto V1 手机 Web 高保真原型

这是第一版产品体验验证用的静态原型，不是正式 ESP32 Web 实现。

## 原型目标

- 手机打开后能快速判断系统是否正常。
- 门控、下料、温湿度、记录、设备维护、微信通知的入口清楚。
- 暂停自动执行、供电恢复、疑似异常等关键状态能被用户看懂。
- 手动下料、开门、关门、停止等动作有防误触确认。
- 视觉上采用简洁控制台风格，减少粗体、大字号和强色块，避免第一屏压迫感。

## 信息架构

1. 首页
   - 系统总状态
   - 门控状态
   - 下料状态
   - 室外温湿度
   - 最近关键事件
   - 正常和告警两种状态
2. 供电恢复报告
   - 恢复时间
   - 推测断电范围
   - 错过的门控/下料计划
   - 自动计划是否恢复
3. 门控
   - 每日多个计划点
   - 开门、关门、停止
   - 暂停自动门控
   - 暂停期间说明：允许手动，错过计划不补跑
4. 下料
   - 每日多个计划点
   - 每个计划配置各下料器投喂量
   - 暂停自动下料
   - Web 手动实时下料
5. 设备维护
   - 分站列表
   - 设备类型
   - 通讯、电机、电流、编码器摘要
   - 设备详情
6. 记录
   - 门控记录
   - 下料记录
   - 系统记录
   - 设备维护记录
7. 微信通知
   - 通知开关
   - 告警类型
   - 测试通知
8. 温湿度
   - 当前温湿度
   - 今天、30 天、1 年趋势
   - 历史记录
   - 采样和长期保存策略
9. 更多
   - 分类记录
   - 温湿度历史
   - 供电恢复报告
   - 设备维护
   - 微信通知
   - 原型状态入口

## 覆盖页面/状态

- 正常首页
- 有告警首页
- 供电恢复报告
- 门控计划页
- 门控暂停中
- 下料计划页
- 下料暂停中
- Web 手动下料
- 温湿度历史页
- 设备维护列表/设备详情
- 分类记录页
- 微信通知设置/测试通知

## 交互约定

- 顶部状态条展示当前页面的关键系统状态。
- 手机底部使用 5 个正式入口：首页、计划、手动、温湿度、更多。
- “更多”承载完整导航，进入记录、维护设置、供电恢复报告、微信通知和原型状态。
- 桌面端保留左侧完整导航，便于查看和审查所有页面。
- 危险或立即动作采用“两步确认 + 长按发送”。
- 暂停只影响自动执行，不修改长期计划；暂停期间允许手动；错过计划默认不补跑。

## 当前视觉原则

- 首页用“运行总览/告警总览”承载判断，不用大面积宣传式标题。
- 常规信息不用加粗，只有品牌和危险/主动作按钮轻微强调。
- 正常、告警、暂停状态主要通过边线、浅底色和小状态标识区分，避免大面积强色块。
- 手机下首页状态卡保持两列，提高扫描效率；极窄屏幕才退回单列。
- 配色以冷灰、青绿色和低饱和蓝色为主，告警色只用于需要注意的状态。
- 日常用户页使用用户语言：门、下料、温湿度、设备连接、微信提醒。
- 维护页保留工程语言：RS485、分站、通讯、编码器、电流、地址。

## 温湿度记录策略

- 第一版原型建议每小时采样一次，日常趋势足够用，也更适合长期保存多年。
- 每天约 24 个原始点，每年约 8760 个原始点。
- 长期保存建议保留每小时原始点，同时生成每日最高、最低、平均值用于快速查询。

## 设备维护分组

- 设备维护按类型分组，不把所有 RS485 设备混在一个连续编号里。
- 门控设备使用门控 01、门控 02。
- 下料设备使用下料 01、下料 02、下料 03、下料 04。
- 传感器使用温湿度 01。
- RS485 地址、电流、编码器、通讯延迟等工程字段只放在维护页。

## 未进入本原型的事项

- 真实 ESP32 路由、API、持久化结构。
- 实际 RS485 协议字段。
- 微信/Blinker 真实 token 保存流程。
- 精确缺料判断。
- 桌面端复杂管理后台布局。
