# FarmAuto V2 手机 Web 高保真原型

这是基于全局设计审计后重构的 V2 原型，不是正式 ESP32 Web 实现。

## 为什么重做 V2

V1 后期进入了错误工作模式：用户指出一个点就修一个点，导致原型逐渐变成局部补丁集合。V2 改为先做整体产品判断，再一次性重构页面、导航和业务闭环。

## 全局设计审计

### 系统性问题

1. 导航不是最终产品导航
   - V1 顶部横向页签像原型目录，不像真实产品。
   - 页面入口、低频功能和维护功能混在一起。
   - 移动端和桌面端导航职责不清。

2. 日常用户页和维护页语言混用
   - 首页出现 RS485、SHT30、分站、编码器等工程词。
   - 用户高频页应该说门、下料、温湿度、设备连接、微信提醒。
   - 工程词只应该出现在设备维护、设置、诊断类页面。

3. 字重和信息层级失控
   - 过多标题、卡片标题和状态值使用加粗，导致页面没有重点。
   - 加粗变成噪声，而不是强调。

4. 温湿度功能被当成事件，而不是长期数据
   - 首页“最近关键事件”里出现温湿度记录，不符合关键事件定义。
   - 温湿度需要独立历史页面，支持长期查看和图表。

5. 设备维护模型不清
   - 所有设备混在一个编号列表里，不利于理解。
   - 应按设备类型分组：门控、下料、传感器。

6. 闭环表达不完整
   - 供电恢复、微信通知、异常记录、暂停自动执行、手动操作之间没有形成完整路径。
   - 用户看到异常后应该知道：发生了什么、是否已通知、是否影响自动计划、下一步做什么、哪里看记录。

### 表层修正但不够的地方

- 降低字重只能改善观感，不能解决信息架构问题。
- 把 RS485 改成设备连接只能改善文案，不能解决维护页归属问题。
- 给温湿度加一个历史页只是开始，还需要长期范围、采样策略和存储表达。
- 给设备页分组只是结构基础，还需要导航、记录和异常路径配合。

## V2 推荐方向

### 核心原则

- 日常优先：正常用户每天打开首页只看状态、下一步计划、关键异常。
- 维护收纳：RS485、地址、电流、编码器、通讯延迟进入维护页。
- 低误操作：手动门控、手动下料、恢复自动计划都需要确认或长按；长按统一为 1 秒。
- 长期运行：温湿度、记录和供电恢复报告按多年无人值守场景设计。
- 异常闭环：异常首页显示问题事实，微信通知、记录、设备维护和暂停状态互相连通。

### V2 导航

手机底部正式导航：

- 首页
- 自动
- 手动
- 记录
- 设置

桌面端使用同样五个正式入口作为左侧主导航，不再展示“正常首页/异常首页/计划总览”等原型目录。

一级入口职责：

- 首页：每天打开先看系统是否正常、下一次自动动作、门控、下料、温湿度和关键事件；新生成的供电恢复报告在首页提示，直到用户选择不再显示。
- 自动：管理门控计划和下料计划；门控计划只支持每天开门/关门，不做周几计划和“复查”动作；暂停只暂停自动执行，不改长期计划。
- 手动：放立即动作，包括门控开/关/停和手动下料；危险动作需要确认或长按 1 秒。
- 记录：查看门控、下料、温湿度、系统事件、设备通讯记录。温湿度历史作为记录分类进入，不占首页或记录页顶部黄金位置。
- 设置：放系统配置、设备维护、微信通知，不放日常操作。

设置页承载：

- 系统设置
- 下料器校准
- 门控校准
- 设备维护
- 微信通知

供电恢复报告不是一级功能。它是一次系统事件的详情：断电恢复后首页提示；查看报告不会自动消失，只有点“不再显示/首页不再提示”才从首页移除；历史报告从“分类记录 > 系统事件”查看。

## 系统设置设计

V2 系统设置页按“可修改项优先、说明信息靠后”组织，避免把不可修改的长期策略放在页面最上方占据主要位置。

参数归属：

- 可修改项：温湿度记录间隔、设备轮询间隔、离线判定次数、时间同步。
- 时间与供电恢复：系统显示 RTC/联网校时状态，可手动同步时间；供电恢复报告在首页提示到用户手动关闭，历史从系统事件记录进入。
- 当前规则说明：错过计划默认不补跑、暂停期间允许手动、供电恢复后恢复后续计划、生成系统记录和微信通知。
- 暂停恢复方式：不放在系统设置里。门控和下料都在执行“暂停自动”时现场选择“暂停到指定时间自动恢复”或“无限期暂停，手动恢复”，并展示暂停期间会跳过哪些计划。
- 下料器校准：每个下料器的“每圈多少克”放在独立校准页；已知参数时直接输入每圈克数，未知时使用“称重标定”：选择下料器、试转固定圈数、称重、计算并保存每圈克数。
- 门控校准：开门位置、关门位置和当前位置放在独立维护页；已知参数时直接输入圈数，未知时使用“行程标定”：按步长移动到关门/开门位置并保存当前位置。需要完整自动学习时称为“行程学习”。当前原型示例为关门 0 圈、开门 5.5 圈。
- 温湿度记录：室外温湿度传感器默认每 1 小时 1 个点，设置范围 1 到 24 小时；长期保存小时原始点，并支持近 24 小时、30 天、1 年范围查看。
- 设备通讯：按门控、下料、传感器分组管理；RS485 地址、通讯延迟、电流、编码器等只在维护页出现。
- 通知与记录：微信通知开关、测试通知、门控/下料/系统事件/设备通讯记录分类。

计划页直接显示两个模块：门控计划和下料计划。计划页只负责“每天哪个设备几点执行什么”：门控计划必须绑定具体门控设备，再设置每天开门/关门时间点；下料计划可以按饲料克数或输出轴圈数设置每个下料器数量，并实时显示另一种单位换算。门控和下料计划都固定为每天执行，不再提供只有一个选项的“重复”下拉框。手动开关门、手动下料不放在计划页。

手动下料支持两种统一输入方式：

- 按重量：每个下料器填写克数，系统按校准值换算圈数。
- 按输出轴圈数：每个下料器填写圈数，最小 0.1 圈，同时显示估算重量。

手动门控必须先选择具体门控设备，再开门、关门或停止；同时显示所选门控的当前门位，例如 5.5 圈、100%。关门位置示例为 0 圈，开门位置示例为 5.5 圈。

当前执行状态：

- 首页和手动页都显示当前执行状态；正常空闲时显示“当前没有正在执行的动作”，不显示停止按钮。
- 门控运行中必须绑定具体门控设备，例如“门控 01 · 主门”，显示动作方向、当前位置、目标位置和进度，并提供停止该门控设备。
- 下料运行中必须绑定具体计划或手动任务，例如“18:00 晚料”，显示本次合计进度，并列出参与的每个下料器及各自进度；无论任务是按重量还是按圈数创建，运行状态固定同时显示克数和圈数；停止按钮停止本次下料任务。

### 温湿度设计

- 默认每 1 小时采样 1 个点，可在系统设置中改为 1 到 24 小时间隔。
- 默认视图显示近 24 小时的 24 个小时点。
- 30 天视图显示近 30 天趋势和统计。
- 1 年视图显示月度趋势和长期汇总。
- 原型默认保存每小时原始点，同时生成每日最高、最低、平均值。

### 设备维护设计

- 门控设备：门控 01、门控 02。
- 下料设备：下料 01、下料 02、下料 03、下料 04。
- 传感器：温湿度 01。
- 维护详情中再显示 RS485 地址、通讯、电流、编码器等工程信息。

## 仍需后续裁决

- 温湿度多年历史的实际存储位置、保留周期和导出方式。
- 门控设备数量是否第一版只启用 1 个，还是预留多个门控。
- 下料器数量第一版是否固定为 4 个，还是允许配置扩展。
- 微信通知实际接入方式和密钥保存策略。
