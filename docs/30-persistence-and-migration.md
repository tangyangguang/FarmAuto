# 持久化、版本与升级原则

## 总原则

用户数据、生产配置、日志、统计、校准参数等持久化资产必须被保护。

涉及持久化结构变化时，必须先给出版本迁移、导出/导入、备份恢复或明确升级路径的方案，让用户确认后再实施。不能静默清空可识别数据。

所有持久化存储都必须考虑磨损均衡。无论使用 AT24C EEPROM、ESP32 flash 文件系统、FRAM、SD 卡或其他介质，方案中都必须说明写入频率、物理写入分散策略、容量告警和远程诊断方式。

AT24C128 适合保存配置、校准、关键状态和小型 wear-levelled record，不适合作为多年原始事件日志的主存储。多年原始记录应单独设计存储介质和数据保留策略。

## 首版历史兼容

FarmAuto 首版不迁移 old_prj 的 AT24C 数据。

这表示：

- 不读取老项目存储格式。
- 不兼容老项目 key 或页槽。
- 不沿用老项目结构。
- 首次使用新固件时，需要明确初始化新格式。

但一旦 FarmAuto 新格式投入使用，后续版本必须保护新格式数据。

## 版本化要求

每类持久化 payload 都必须包含：

- magic。
- schema version。
- payload length。
- CRC。
- 必要的保留字段。

应用读取数据时必须检查：

- magic 是否匹配。
- schema version 是否支持。
- payload length 是否合理。
- CRC 是否正确。
- 字段值是否在有效范围。

## 保存时机

配置数据：

- 用户保存配置时写入。
- 校准参数变化后写入。

运行状态：

- 电机停止后写入。
- 故障停机后写入。
- 归零或清空计数后写入。
- 日期切换后写入。

不建议：

- 电机运行中高频写入。
- 每次编码器脉冲变化都写入。
- 内容没有变化仍写入。

## 初始化和恢复

首次初始化必须是显式行为，至少满足以下之一：

- Web 维护页面二次确认。
- 串口维护命令。
- 固件首次启动检测到完全空白 EEPROM 后，明确记录“初始化新格式”事件。

读取失败时：

- 配置记录无效：使用默认值，但应提示未加载到有效配置。
- 状态记录无效：进入需要校准或维护的状态。
- 历史记录无效：只丢弃无效记录，不影响其他记录。

## 升级策略

后续 schema 变化时，应优先提供迁移：

- 从旧 version 读取。
- 校验字段。
- 填充新字段默认值。
- 写入新 version。
- 保留旧记录直到新记录写入成功。

如果某次变化无法安全迁移，必须在文档中说明原因，并让用户确认清空或重新初始化。

## 备份与诊断

应用应提供维护信息：

- 当前 schema version。
- 当前记录 sequence。
- 最近保存时间。
- CRC 校验状态。
- EEPROM 设备在线状态。
- 写入失败计数。

是否提供导出/导入功能，等进入实现阶段再根据 Web 能力和存储容量评估。
