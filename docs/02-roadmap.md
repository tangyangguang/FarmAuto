# FarmAuto 实施路线图

## 当前状态

FarmAuto 当前处于规划与架构阶段。已经完成：

- 工作区目标和约束。
- Esp32FarmDoor / Esp32FarmFeeder 初步应用边界。
- `docs/apps/` 与 `docs/libs/` 文档分组。
- 三个公共库的初步职责边界。
- 公共库与 Esp32Base 的分工原则。
- 持久化、版本和升级原则。

两个应用项目的具体需求仍需继续沟通完善，暂不进入最终需求冻结和编码阶段。

## 总体阶段

### 阶段 0：规划与架构

目标：

- 明确 monorepo 结构。
- 明确应用和公共库边界。
- 明确 Esp32Base 与 FarmAuto 公共库的分工。
- 明确当前阶段不写源码、不创建 PlatformIO 工程。

当前阶段剩余工作：

- 继续完善两个应用项目的需求设计。
- 明确每个应用的首版验收标准。
- 明确实机实验项和默认参数来源。

完成标准：

- 文档能指导后续创建工程结构。
- 公共库文档不包含具体应用独有逻辑。
- 应用文档只描述应用业务和如何组合公共库。

### 阶段 1：公共库接口方案冻结

目标：

- 冻结 `Esp32EncodedDcMotor` 的接口级设计。
- 冻结 `Esp32MotorCurrentGuard` 的接口级设计。
- 冻结 `Esp32At24cRecordStore` 的接口级设计。
- 在冻结前查阅成熟 Arduino/PlatformIO 库和芯片官方资料，确认接口风格、非阻塞模型、芯片配置方式和独立库结构。

产物：

- 每个公共库的配置结构。
- 每个公共库的状态结构。
- 每个公共库的错误码或故障原因。
- 与上层项目的组合方式。
- 首版只实现哪些能力、明确不实现哪些能力。

完成标准：

- 不需要了解 Esp32FarmDoor 或 Esp32FarmFeeder 的业务细节，也能独立实现公共库。
- 公共库核心逻辑不强绑定 Esp32Base。
- 芯片支持范围清楚，未来扩展路径清楚。
- 公共库按未来独立仓库标准设计，但当前不一次性拆出三个独立仓库。

### 阶段 2：创建公共库骨架

目标：

- 创建 `lib/Esp32EncodedDcMotor/`。
- 创建 `lib/Esp32MotorCurrentGuard/`。
- 创建 `lib/Esp32At24cRecordStore/`。

产物：

- 最小 Arduino/PlatformIO library 结构。
- README 或库内说明。
- 头文件和源文件骨架。
- 基础示例或测试入口。

完成标准：

- 库可以被后续应用项目引用。
- 不包含自动门、喂食器等业务代码。
- 不修改 Esp32Base。

### 阶段 3：公共库实现与验证

目标：

- 实现可独立验证的公共能力。

建议顺序：

1. `Esp32At24cRecordStore`
2. `Esp32EncodedDcMotor`
3. `Esp32MotorCurrentGuard`

原因：

- 持久化库会影响两个应用的数据格式和升级策略。
- 电机库是两个应用的运动控制核心。
- 电流保护库依赖实机采样和阈值实验，接口可先冻结，实现可分阶段完善。

完成标准：

- `Esp32At24cRecordStore` 能完成基本读写、CRC、双槽或追加式最新记录选择、写前比较。
- `Esp32EncodedDcMotor` 能完成软启动、软停止、目标脉冲、状态机、基础保护。
- `Esp32MotorCurrentGuard` 能完成 INA240A2 电流换算、滤波、启动宽限、连续超限判定。

### 阶段 4：Esp32FarmDoor 首版

目标：

- 实现自动门本地可靠控制。

进入条件：

- Esp32FarmDoor 需求已沟通确认。
- 硬件参数和默认引脚确认。
- 端点校准、停止、电流保护、持久化策略确认。

首版能力方向：

- 本地 Web 控制。
- 物理按钮。
- 编码器位置。
- 开门/上限位防止过度卷绳。
- 端点校准二次确认。
- INA240A2 电流保护。
- AT24C 持久化。
- 关键事件记录。

完成标准：

- 能实机开门、关门、停止。
- 故障时能安全停机。
- 断电重启后能恢复关键配置和位置。
- Web 能查看状态并保存配置。

### 阶段 5：Esp32FarmFeeder 首版

目标：

- 实现三路喂食器本地可靠控制。

进入条件：

- Esp32FarmFeeder 需求已沟通确认。
- 3 路电机参数和下料标定策略确认。
- 电流检测硬件通道数确认。
- 今日累计、长期原始记录和导出策略确认。

首版能力方向：

- 单路启动/停止。
- 启动全部/停止全部。
- 克数/圈数目标模式和内部脉冲换算。
- 每路饲料桶容量和余量配置。
- 今日累计。
- 长期原始记录。
- 清空当天计数二次确认。
- 电流保护预留策略，当前硬件无电流检测。

完成标准：

- 3 路喂食动作可控。
- 今日累计和历史记录可靠。
- 断电重启后统计和配置可恢复。
- 异常时能按策略停机。

### 阶段 6：实机校准与发布收口

目标：

- 将默认参数从假设值调整为实测值。
- 收敛故障策略和维护流程。

需要确认：

- Esp32FarmDoor 正常开门/关门电流。
- Esp32FarmDoor 堵转或接近堵转电流。
- INA240A2 零点和 ADC 偏差。
- Esp32FarmFeeder 每路每圈下料克数。
- Esp32FarmFeeder 多路启动电流，当前阶段只能通过外部仪表实测。
- 软启动和软停止默认时长。
- AT24C 写入频率是否合理。

完成标准：

- 默认配置可用于真实设备。
- 故障阈值有实测依据。
- 文档和代码保持一致。
- 发布前检查清单完成。

## 当前优先事项

下一步应继续完善两个应用项目的需求设计，但不要急于编码。

建议沟通顺序：

1. Esp32FarmDoor：门的完整状态机、端点校准策略、停止策略、电流保护策略、Web 页面需求。
2. Esp32FarmFeeder：三路运行模式、电流检测通道、投喂目标用克数还是圈数、历史统计格式、清计数策略。
3. 三个公共库：接口级方案冻结。

在应用需求未确认前，可以继续完善公共库文档，但不应开始应用代码实现。
