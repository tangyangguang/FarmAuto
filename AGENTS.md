# FarmAuto 项目规则

## 沟通

- 默认使用中文，表达直接、清晰、务实。
- 用户提出想法时先帮助澄清和完善方案，不急着写代码。
- 需要专业判断时，主动给出推荐和理由，不把所有技术决策都推给用户。
- 每次回复说明已完成内容、验证情况和剩余工作。

## 项目目标

- 项目重写为 ESP32 主控 + RS485 分站架构。
- 主控负责联网、Web、配置、调度、记录和分站管理。
- 分站负责本地闭环控制电机、编码器、电流检测和安全停止。
- 电机动作必须由分站本地闭环完成，主控不依赖实时通信决定停止时机。

## 硬件边界

- `old_prj/` 是正在运行的老项目，只读参考，禁止修改、删除或提交。
- `pcb/` 是 PCB、BOM、网表资料，只读参考，修改前必须得到用户明确确认。
- `_pending_delete/` 是旧实现归档，默认不维护、不参考，除非用户明确要求。
- 主控板使用 ESP32、RS485、RTC、FRAM、4 个按钮、GPIO0 长按功能和 2 个 LED。
- 主控板不使用屏幕、I2C 电平转换、干接点输出和直接电机控制。
- 分站板目标为 STC8H8K64U、RS485、拨码地址、AT8236、编码器和 INA240A2。

## 软件边界

- 分站分为下料模式和门控模式，但硬件板尽量统一。
- 下料模式支持按圈数或克数下料，每圈克数可配置。
- 门控模式支持开门、关门、停止、位置校准和异常保护。
- 电流检测用于过流、堵转、异常辅助判断；缺料检测不纳入当前成品需求。
- 主控复杂配置放 Web，本地按钮只做简单应急操作。

## 工作方式

- 新增大功能、协议、持久化结构、硬件方案前，先给方案并等待确认。
- 修改文件前先理解影响范围，不覆盖用户已有改动。
- 不提交密钥、WiFi 密码、Blinker token 或其他敏感信息。
- 完成明确阶段后再考虑提交和推送；提交前必须检查 `git status`。

## 轻量原型与小程序开发模式

当用户明确要求创建、重做、修改、实现、修复原型、小程序、页面或工具时，默认进入执行模式，不要一步一问。

- 除非缺失信息会导致明显方向错误、高影响操作、不可逆修改或安全风险，否则先基于合理假设完成一版可运行结果。
- 不要默认写大型计划、详细规格文档或长篇方案；必要计划控制在 3-5 步以内。
- 不要默认使用 TDD；早期只做语法检查、smoke test、关键交互验证和必要的核心规则测试。
- 不要默认启用多子代理长期并行开发；子任务必须短、窄、可验收。
- 不要提前抽象 shared、framework、base、platform 层；只有真实重复出现后再考虑抽取。
- 每轮优先交付用户能看到、能运行、能点击或能验证的垂直闭环。
- 原型阶段先确认页面结构、信息层级、字段、状态和流程，再打磨视觉样式。
- 信息不足时，在最终回复中列出“本轮假设”和“后续需要确认点”，不要因此频繁中断。
- 如果发现当前方向会导致复杂化、代码膨胀或返工，应主动停下说明原因，并给出更简单方案。
